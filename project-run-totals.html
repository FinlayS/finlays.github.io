Run Totals Web Application

Web application project.


An application that calculates the total "active" distance and pace for a running activity with multiple split laps of effort and recovery.


I undertook the task of learning to build a fully functioning web application to address a couple of points:

  • to scratch a nerdy itch to know whether the running sections of my regular run/walks were getting longer and faster.
  • a desire to better understand the potential overlap between the different types of automated testing.

Having learned to code by writing tests and having been in the position in the past of making the mistake of maintaining a test suite that tests the bulk of the acceptance criteria through the UI, I knew that learning to build an application of this sort would help me avoid this mistake again.

What did I learn in the process? 

I'm still a tester at heart

Much as I learned a lot and enjoyed the experience of developing an application, and still want to carry on improving my coding skills, I didn't come away feeling like I now want to cross to the other side. Although I've now blurred the lines a bit, I still feel that having "tester" as my unique selling point within the team is still valid.

One day all devs will be testers

...perhaps. As I said earlier, it did blur the lines for me between an SDET and a developer in the workplace. With the right mindset and culture within the team I see no reason why the two roles should not eventually become one and the same.

I've come to the conclusion that one of my goals at work should be to make the role of test engineer irrelevant.

The test pyramid is more than just words

Ok, so I wholly understood the concept of the test pyramid before starting this project, but I've always been a visual kind of person who needs to see things to understand them and how they work.

Undertaking this project really has helped me understand the different parts of the application under test and where each test suite plays it's part in testing and where any potential overlap or duplication may arise.

Have that conversation

This exercise wasn't just borne from a desire to avoid a bloated slow UI test suite as part of a build process, It also taught me that lack of conversation was one of the things that put me in that position in the first place.

Nowadays the team regularly discusses and reviews the test process so we all have an input of view of it's vision and purpose.
